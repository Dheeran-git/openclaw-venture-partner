# OpenClaw Venture Partner — Project Instructions for Claude Code

You are the engineering collaborator on **OpenClaw Venture Partner**, an autonomous AI deal-flow agent for freelancers and digital agencies. This document is your operating manual. Read it fully on every session. Do not deviate from it without explicit user permission.

---

## What we're building

OpenClaw Venture Partner is an AI agent that runs the full deal lifecycle for solo operators and small agencies:

1. **Finds leads** across three "layers" of demand:
   - **Layer 1 (Scout):** explicit demand — job postings on Upwork, LinkedIn, Freelancer, Indeed, etc.
   - **Layer 2 (Architect):** warm leads from growth signals (funding rounds, new hires, site issues), with auto-generated proof-of-value (sample components, Lighthouse audits, bug reports) attached to outreach.
   - **Layer 3 (Negotiator):** real-time guidance on client conversations + persistent memory across the relationship for upsells.

2. **Drafts pitches** personalized to each lead.
3. **Routes everything through human approval gates** — the agent never sends, posts, or commits anything without explicit user approval. Draft-only is the brand promise.
4. **Remembers every client** in persistent markdown-style memory.

The product surface is a **web dashboard** plus chat-platform interfaces (Telegram, Discord, WhatsApp, Slack) for on-the-go approvals.

## Build context: hackathon-shaped, production-aligned

This is being built for a hackathon. **Speed and demo polish matter.** But every architectural choice should make later production migration cheap, not expensive. Specifically:

- Multi-tenant data model from day one (every table has `user_id`), even though we're hardcoding a single user for the demo.
- Provider abstractions at every external boundary (LLM, scraping, email) so swapping vendors later is a one-file change.
- Real database, real schema, real types — even with one user.
- Real job-queue patterns — even if a single worker for the hackathon.
- No real auth, no Stripe, no onboarding flow, no marketing site for now. Those are additive.

The rule: **cut features and polish, never cut architecture.**

## Tech stack (locked, do not propose alternatives unless asked)

- **Monorepo:** pnpm workspaces. Folders: `apps/`, `packages/`.
- **Frontend:** Next.js 15 (App Router) + TypeScript + Tailwind CSS. Deploy target: Vercel.

- **LLM:** provider-agnostic via an internal client (see "LLM client" section below). Hackathon primary: GitHub Copilot Pro via OAuth (using a dummy GitHub account, never the user's main). Fallbacks: OpenRouter, Google Gemini. Production target: Anthropic API direct.
- **Scraping:** Zyte (free via dummy GitHub Student account) wrapped behind a `Scraper` interface. Fallback: self-hosted Firecrawl or Playwright.
- **Email sending:** Resend.
- **Chat platforms:** Telegram first (best OpenClaw integration). Discord and WhatsApp are listed as "also supported" in the architecture but not demoed.
- **Agent runtime:** OpenClaw (https://github.com/openclaw/openclaw) for the Telegram adapter and the agent loop. Custom skills live in `apps/agent/skills/`.

## LLM client — the most important abstraction

We use multiple LLM providers, with the active one decided at runtime. **Every LLM call in the project goes through this single client.** This insulates us from provider failures, ToS issues, and lets us swap to Anthropic API for production with one config change.

### Where it lives

packages/agent/src/llm/
├── client.ts          # Public interface — what business logic imports
├── types.ts           # Request/response types, Zod schemas
├── router.ts          # Decides which provider to use at runtime
├── logger.ts          # Logs every LLM call to the llm_calls table
└── providers/
├── copilot.ts     # GitHub Copilot OAuth (hackathon primary)
├── openrouter.ts  # OpenRouter (fallback)
├── gemini.ts      # Google Gemini (second fallback)
└── anthropic.ts   # Anthropic API (production target — implement, leave inactive)

### The interface

```typescript
export interface LLMClient {
  complete<T>(opts: {
    user_id: string;
    purpose: 'score_lead' | 'draft_pitch' | 'draft_reply' | 'extract_lead' | 'memory_update';
    prompt: string;
    prompt_version: string;
    schema?: ZodSchema<T>;
    model?: 'fast' | 'balanced' | 'capable';  // Provider maps this to its actual model name
  }): Promise<T>;
}
```

### Rules

- **Every call is logged** to the `llm_calls` table with: user_id, purpose, prompt_version, model, input/output tokens, cost (when available), duration, request, response.
- **Structured outputs are mandatory** when the response is consumed by code (scoring, drafting). Pass a Zod schema; the client validates and retries on failure.
- **The router picks providers in this order:** Copilot if `COPILOT_TOKEN` set and healthy → OpenRouter → Gemini → Anthropic. Health-check on first call per session; cache result for 60 seconds.
- **Never hardcode a provider** in business logic. Business code only ever imports `client.ts`.

### Provider notes

- **Copilot adapter:** uses GitHub OAuth token from `.env` as `COPILOT_TOKEN`. Authenticates against Copilot's chat completions endpoint. **This is a workaround that violates GitHub's ToS for non-coding use cases. Use only for hackathon demo. Production must switch to Anthropic.** The user is aware.
- **OpenRouter:** standard API key in `.env` as `OPENROUTER_API_KEY`. Free tier credits.
- **Gemini:** standard API key as `GEMINI_API_KEY`. Free tier.
- **Anthropic:** `ANTHROPIC_API_KEY`. Implement the adapter, leave it as fallback after Copilot/OpenRouter; activated by setting the env var.

### Build the LLM client FIRST, before any business logic

This is the most important rule in the project. Do not write the scout, the scoring, the drafting, or anything else until the LLM client abstraction is built and tested with all four provider adapters. A 30-minute investment here saves the project if Copilot breaks during the demo.

## Database schema (Supabase Postgres)

Every table has `user_id`. Every query filters by it. RLS policies are NOT enforced for the hackathon — we trust the application layer. After the hackathon, we'll add RLS as the production migration.

### Tables

```sql
-- Profiles (extends Supabase auth.users)
create table profiles (
  id uuid primary key references auth.users(id),
  display_name text,
  skills jsonb,
  hourly_rate numeric,
  portfolio_url text,
  bio text,
  created_at timestamptz default now()
);

create table sources (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references profiles(id),
  type text not null,           -- 'upwork', 'linkedin', etc.
  config jsonb,
  last_scraped_at timestamptz,
  created_at timestamptz default now()
);

create table leads (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references profiles(id),
  source_id uuid references sources(id),
  layer int not null,           -- 1, 2, or 3
  raw jsonb not null,
  normalized jsonb not null,
  hash text unique not null,    -- for dedup
  scraped_at timestamptz default now()
);

create table scores (
  id uuid primary key default gen_random_uuid(),
  lead_id uuid not null references leads(id),
  score int not null,
  reasoning text,
  prompt_version text not null,
  model text not null,
  created_at timestamptz default now()
);

create table pitches (
  id uuid primary key default gen_random_uuid(),
  lead_id uuid not null references leads(id),
  user_id uuid not null references profiles(id),
  draft text not null,
  subject text,
  status text not null default 'draft',  -- draft, approved, sent, rejected
  approved_at timestamptz,
  sent_at timestamptz,
  created_at timestamptz default now()
);

create table clients (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references profiles(id),
  company_name text not null,
  contact_email text,
  source_lead_id uuid references leads(id),
  status text default 'active',
  memory_md text,
  created_at timestamptz default now()
);

create table approvals (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references profiles(id),
  action_type text not null,
  payload_hash text not null,
  resource_type text,
  resource_id uuid,
  status text not null,
  decided_at timestamptz,
  created_at timestamptz default now()
);

create table llm_calls (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references profiles(id),
  purpose text not null,
  prompt_version text,
  model text not null,
  provider text not null,
  input_tokens int,
  output_tokens int,
  cost_usd numeric,
  duration_ms int,
  request jsonb,
  response jsonb,
  created_at timestamptz default now()
);
```

## Architecture patterns

### Event-driven from day one

API routes accept user actions and enqueue events. Workers process events. The dashboard reflects state from the database. Pattern:

Dashboard click "Run scout"
→ POST /api/scout
→ Inngest event "scout/requested"
→ Worker: scrape → score → write to DB
→ Realtime subscription updates dashboard
→ Telegram notification fires

This is the same pattern at 1 user and 10,000 users. No refactoring needed for scale.

### The HITL approval flow (the demo's centerpiece)

This must be built carefully:

1. Agent drafts a pitch. Saves to `pitches` table with `status='draft'`. Computes `payload_hash`.
2. Notifier sends Telegram message + dashboard real-time update with [Approve] / [Edit] / [Reject] options.
3. User clicks Approve (in Telegram or dashboard).
4. Approval handler:
   - Verifies user owns the resource (matches `user_id`).
   - Verifies `payload_hash` at approval time matches the current `pitches.draft` hash (prevents man-in-the-middle swap).
   - Writes `approvals` row.
   - Fires `pitch/approved` event.
5. Worker picks up `pitch/approved`, sends email via Resend, updates `pitches.status='sent'`.
6. Dashboard updates in real-time. Telegram replies "Sent ✅".

Every approval is logged in `approvals` for audit + future training data.

### OpenClaw integration

OpenClaw runs as a separate process. It owns:
- The Telegram adapter (and Discord/WhatsApp later).
- The agent reasoning loop.
- Loading custom skills from `apps/agent/skills/`.

Custom skills are markdown files describing capabilities (scout, score, draft pitch, etc.) that OpenClaw routes user messages to. Skills can call into our `packages/agent` code.

The dashboard, worker, and OpenClaw all read/write the same Supabase database. **The database is the source of truth.** Components don't call each other directly.

### Scraper interface

```typescript
export interface ScrapedLead {
  source_url: string;
  title: string;
  description: string;
  posted_at: Date;
  raw: unknown;
}

export interface Scraper {
  name: string;
  scrape(query: string, limit: number): Promise<ScrapedLead[]>;
}
```

`packages/scraping/src/zyte.ts` implements this for the hackathon. Add `firecrawl.ts` and others later. Business logic only imports the interface.

### Prompts as files

Every prompt lives in `packages/agent/src/prompts/` as a markdown file. Frontmatter declares the model preference, version, and expected schema. Never inline prompts in code.

```markdown
---
version: "v1"
model: "balanced"
schema: "ScoreLeadResponse"
---

You are an expert at scoring freelance leads...

[examples here]

Lead: {{lead_json}}
Profile: {{profile_json}}

Return JSON: {score: number, reasoning: string}
```

The LLM client renders the template, calls the provider, validates the response against the Zod schema named in frontmatter.

## Build order (do not deviate)

Build in this order. Each step must work end-to-end before moving to the next.

### Phase 1 — Foundations (Day 1)

7. Port `Sidebar.jsx`, `Topbar.jsx`, and the empty `LeadTable.jsx` shell from the design system into `apps/web/components/` (as TSX). Render them on a single dashboard page with mock data.
8. Initial commit, push.

### Phase 2 — Scout end-to-end (Day 2)

1. Build the `Zyte` scraper adapter behind the `Scraper` interface. Test against Upwork or Indeed (Zyte handles auth/proxies).
2. Build the lead-scoring prompt and Zod schema. Use few-shot examples in the prompt.
3. Wire as an Inngest function: `scout/requested` event → scrape → score → write `leads` and `scores` rows.
4. Add "Run scout" button to dashboard. Wire to API route → Inngest event.
5. Subscribe the dashboard to Supabase Realtime on the `leads` table. New leads appear without refresh.
6. Adapt the design system's `LeadTable` and `LeadDetail` components to render real data.
7. Telegram bot setup via OpenClaw. Send a digest message: "5 new leads, top scored 92."

### Phase 3 — Pitch drafting + HITL (Day 3)

1. Build the pitch-drafting prompt + schema.
2. Add "Generate Pitch" action on a lead. Wire to API → Inngest event → drafted pitch saved to `pitches`.
3. Adapt the `PitchCard` component to render the draft + approval bar.
4. Build the approval flow: dashboard "Approve & send" button + Telegram inline keyboard.
5. Implement `payload_hash` verification. Log approval in `approvals` table.
6. On approval, fire `pitch/approved` event. Worker sends via Resend (or stubs the send for the demo).

### Phase 4 — Layer 2 proof-of-value (Day 4)

This is the demo's "wow" moment. Pick one:

- **Lighthouse audit pitch.** Worker runs Lighthouse on a prospect's site → generates summary → drafts pitch with the audit attached.
- **Sample component pitch.** Worker generates a small React component matching the prospect's stated tech needs → drafts pitch with code attached.

Lighthouse is faster to implement; lean toward it unless time permits both.

### Phase 5 — Layer 3 negotiation (Day 5)

1. Build reply-drafting prompt that takes incoming email + client memory + outputs three options.
2. Mock an incoming email arrival (for the demo).
3. Adapt the design system to render three reply options + approval.
4. Implement `clients.memory_md` write-on-update.
5. Render the markdown memory in the dashboard.

### Phase 6 — Polish (Day 6)

1. **Demo mode** — a single button that runs a canned end-to-end scenario with predictable data. Critical for the live demo.
2. Empty states for every list.
3. Loading skeletons (use the `component-skeleton-empty.html` reference from the design system).
4. Error states.
5. Deploy to Vercel + Supabase production. Custom domain if available.
6. Backup demo video recorded.

### Phase 7 — Submit (Day 7)

1. Final demo rehearsal × 5.
2. README polish with architecture diagram.
3. Submit.

If any phase is running long, stub the next one rather than skipping foundations. The HITL approval flow and the dashboard are the demo's spine — never compromise them.
